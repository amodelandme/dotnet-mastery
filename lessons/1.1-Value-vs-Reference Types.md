# Value vs Reference Types

> **One-line summary:** A variable either holds the thing itself (value type) or a pointer to the thing (reference type) — and every assignment, method call, and equality check in C# is a consequence of that single distinction.

## The mental model

Assignment in C# always copies **what's in the variable slot**. For a value type, that slot *is* the data. For a reference type, that slot is a pointer to data on the heap.

```
  Struct (value semantics)           Class (reference semantics)
  ────────────────────────           ───────────────────────────

         Stack                          Stack          Heap
     ┌──────────┐                   ┌────────┐     ┌──────────┐
     │ sA: X=1  │                   │  cA ───┼───► │          │
     ├──────────┤                   ├────────┤     │  X = 99  │
     │ sB: X=99 │                   │  cB ───┼───► │          │
     └──────────┘                   └────────┘     └──────────┘

     independent copies             two labels, one object
     prints: 1, 99                  prints: 99, 99
```

Mutation follows the arrow. For a struct, there is no arrow — `sA` and `sB` are the data. For a class, mutating through any reference to an object is visible through every other reference to the same object.

## Core rules

1. **Assignment copies the slot.** Structs → two independent copies. Classes → two pointers to one object.
2. **Mutation through a reference is shared.** Anyone holding a reference to that object sees the change.
3. **C# is always pass-by-value.** For reference types, the *value* being passed is the reference. Reassigning the parameter inside a method only moves your local copy — the caller still points at the original object. Mutating through the parameter changes the shared object, and the caller sees it.
4. **"Structs live on the stack" is a useful lie.** A struct lives wherever its container lives — a struct field inside a class sits on the heap; a captured struct in a lambda goes in the closure on the heap; a boxed struct lives in its box on the heap.
5. **`record` turns on value equality.** For both `record class` and `record struct`, the compiler auto-generates `==`, `Equals`, and `GetHashCode` that compare by all properties. `ReferenceEquals` is the escape hatch that ignores the override.
6. **Modern C# refuses the silent-copy bug.** `foreach (var p in listOfStructs) p.X = 99;` is rejected as **CS1654** — the iteration variable is a copy, mutation would silently do nothing, so the compiler won't let you write it.

## Minimal example

```csharp
struct PointS { public int X; }
class  PointC { public int X; }

var sA = new PointS { X = 1 };
var sB = sA;
sB.X = 99;                              // sA.X still 1 — independent values

var cA = new PointC { X = 1 };
var cB = cA;
cB.X = 99;                              // cA.X now 99 — same object

public record class  PointRC(int X);
public record struct PointRS(int X);

new PointRC(42) == new PointRC(42);                  // True  (value equality)
ReferenceEquals(new PointRC(42), new PointRC(42));   // False (distinct heap objects)
```

## Gotchas

- **`foreach` on a `List<struct>`** can't mutate elements (CS1654). To update in place, replace the element: `list[i] = new Point { X = 99 };`.
- **Arrays of structs behave differently from lists.** `array[i].X = 99` works because array indexers return `ref` to the element; `List<T>`'s indexer returns a copy. Subtle consistency trap.
- **Reassigning a class parameter inside a method doesn't escape the method.** Use `ref PointC p` if you want the caller to see the new reference. Otherwise, mutate through the parameter.
- **Boxing is invisible and allocating.** Storing a value type in an `object` or non-generic interface silently allocates a new heap object each time. Hot loops pay the GC tax.
- **Records don't fix mutable internals.** A record with a `List<T>` property compares the list *reference* (via `EqualityComparer<T>.Default`, which for `List<T>` is reference equality). Two "equal" records can diverge later if one of those lists is mutated.

## Interview questions this answers

- *"What's the difference between a struct and a class?"* → value vs reference semantics; copy behavior on assignment; mutation propagation; where they live.
- *"Is C# pass-by-value or pass-by-reference?"* → always pass-by-value. For reference types, the value being copied *is* the reference — so you get pass-by-value-of-a-reference.
- *"When would you choose a struct over a class?"* → small (≤ ~16 bytes), immutable, value-like data (`Point`, `Money`, `Duration`); performance-critical paths where GC allocation matters.
- *"What's the difference between a `class` and a `record class`?"* → `record class` auto-generates value-based `==`, `Equals`, `GetHashCode`, `ToString`, plus `with`-expressions. Great for DTOs and value objects.
- *"What does `record struct` give you that a struct doesn't?"* → auto-generated value equality without having to implement `IEquatable<T>` by hand.
- *"Why can't I mutate a struct in a `foreach` loop?"* → CS1654. The iteration variable is a copy of the element; modern C# refuses to let you write the silent-no-op bug.
